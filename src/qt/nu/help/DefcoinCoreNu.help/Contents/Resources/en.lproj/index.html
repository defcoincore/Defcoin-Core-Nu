Home (Overview)

Defcoin Core Nu is a native full-node wallet for the Defcoin network. It validates blocks, relays peer-to-peer traffic, stores wallet data locally, signs transactions locally, and gives the user direct control over receiving, sending, backup, encryption, and node operation.

Nu separates the presentation layer from the inherited Litecoin-derived backend. The backend owns consensus, wallet storage, private keys, transaction signing, peer networking, and RPC. The Qt Quick frontend uses local RPC and keeps irreversible wallet actions explicit.

Use Defcoin Core Nu when you want the wallet and validating node on your own computer. The wallet pages manage keys and transactions. Diagnostics shows synchronization, peer connectivity, network traffic, local logs, RPC console output, and peer inspection.

Create or open wallets carefully. Back up wallet.dat before upgrades, imports, rescans, recovery work, or moving to another computer. If the wallet is encrypted, keep the passphrase offline and test backups before relying on them.

Cryptocurrency Basics

A cryptocurrency wallet does not store coins as files inside the app. It stores keys that can authorize transactions on the network. Your balance is calculated from the blockchain: the shared transaction record that full nodes validate and keep in sync.

A full-node wallet such as Defcoin Core Nu downloads and verifies block and transaction data itself. That takes more disk, network, and startup work than a lightweight wallet, but it gives stronger local verification and better visibility into what your node is doing.

A transaction spends previously received outputs to one or more new destinations. A receive address is like an invoice destination: it can be shared so another wallet can pay you. A transaction ID is the public identifier for a broadcast transaction. Confirmations count how many blocks have been added after the transaction was included in a block.

Fees pay miners to include transactions in blocks. Higher fees may confirm faster when the network is busy; lower fees may take longer. Nu shows fee choices before broadcast because sending is a one-way action once the backend accepts and broadcasts the transaction.

Peers are other nodes your node talks to. They relay blocks, transactions, and peer addresses. More peers can improve network visibility, but peer information is not proof of identity.

Status And Indicators

The Home status strip is intentionally short. It shows the current network state, peer count, block height, synchronization state, and wallet lock state without repeating the same message in several places.

Network connected means the backend reports active P2P networking and at least a usable node state. Network isolated means peer networking is disabled or not yet active. Settings > Network can request Connect or Isolate.

Wallet locked means spending/signing requires unlocking the wallet. Unlock only when spending or signing.

Send

Send requires a recipient, amount, and fee choice. Use File > Open URI for defcoin: payment links, or paste/type the recipient directly. Use labels and the address book when available to reduce paste errors. Review every transaction before broadcast because blockchain transactions are intentionally hard to reverse once confirmed.

Fee controls are inherited from the backend. If a recommended fee cannot be estimated, the interface should make that state clear instead of pretending the recommended option changed the fee.

Advanced send options include custom change-address entry and PSBT tools. PSBT tools can create a PSBT from the current Send form, load a PSBT from file or clipboard, ask the wallet to sign available inputs, finalize a complete PSBT, broadcast the finalized transaction, copy the current PSBT, save it, or clear it from the interface.

Partially Signed Transactions

PSBT means Partially Signed Bitcoin Transaction. Defcoin inherits this transaction handoff format from Bitcoin-family wallet code. A PSBT is not a normal sent payment yet. It is a package of transaction information that can be reviewed, signed, passed to another signer, finalized, and then broadcast.

New users usually do not need PSBT. Use the normal Send flow when this wallet has the keys and you simply want to pay someone. PSBT is useful when the signing step is separated from the preparation step, such as an offline signing machine, a watch-only wallet, a hardware signer, or a multi-person review process.

Before using PSBT, make sure you know which wallet owns the keys, which device is expected to sign, and which device is expected to broadcast. If you are unsure, do not finalize or broadcast until the transaction has been reviewed by someone who understands the signing setup.

Wallet Safety

  • Encrypt wallets that hold meaningful value.
  • Keep multiple offline backups of wallet.dat after creating or importing keys.
  • Do not paste commands from strangers into Console.
  • Do not expose private keys to miners, pools, websites, or support chats.
  • Test restores with a copy before depending on a backup.

Receive

Receive generates a receiving address and QR payment request. The QR code encodes the request information for easier transfer to another wallet. A receive request is local wallet metadata and does not itself prove that money was sent.

Use the row information action to reopen a saved receive request, copy its address or URI, and view the QR code. Receiving does not require the sender to know your wallet file or private keys.

Activity (Transactions)

Activity lists transactions with sortable columns. Columns should sort by their real data type: dates as dates, amounts as numbers, and addresses as addresses rather than plain alphabetic strings. Column widths can be dragged and reset from Settings.

Use the information action on a transaction row to open details such as status, confirmations, transaction ID, and explorer links when third-party explorer URLs are enabled. Text can be selected and copied from the details window.

Export CSV when you need an external accounting or review copy. CSV exports are plain text and may reveal transaction history.

Diagnostics

Status summarizes client version, user agent, data directory, connections, mempool size, chain height, verification progress, network activity, traffic totals, wallet state, and backend hydration.

Log shows debug.log lines since this Nu launch in local time. Open Full Log opens the complete debug.log in the operating system's normal viewer. The full debug.log keeps the backend's Litecoin-compatible UTC timestamp format.

Console runs local RPC commands and keeps command output visible so prior responses can be copied into later commands. Do not paste commands from strangers into Console. RPC commands can change wallet or node state.

Traffic charts recent received and sent network rates and shows total traffic received and sent. Peaks are diagnostic: a high receive spike often means block or transaction data arrived; a high send spike means this node sent data to peers.

Peers lists connected peers with Node Id., direction, IP address, port, ping timing, sent and received totals, user agent, magic bytes, protocol version, and service flags. The Magic column is read from the actual P2P packet header selected for that peer, not inferred from user-agent text. fbc0b6db means the peer is still using the legacy Litecoin-compatible Defcoin bytes. defc014e means the peer is using the new Defcoin message bytes. IPv4 addresses use right-aligned octets; IPv6 addresses keep their compressed display form and sort numerically. For inbound TCP connections, the displayed port can be the remote source port used for that session, not necessarily the remote node's listening port.

Explicit addnode works when the exact host and port are known. Automatic discovery depends on DNS seeds, fixed seeds, or normal peer address gossip advertising the exact endpoint. A DNS A or AAAA record only gives an address, not a non-default Defcoin port.

The Only accept Defcoin-prefixed User Agents setting rejects peers whose advertised user agent does not begin with Defcoin, ignoring case. This improves network hygiene, but it may reduce discovery if older nodes relay useful addresses while identifying themselves with legacy Litecoin-style user agents.

Reverse DNS, LAN names, geolocation, and autonomous-system data are best-effort diagnostics, not identity proof.

Unsupported Or Compatibility-Limited Features

Defcoin mainnet does not currently treat SegWit, CSV, CLTV, BIP66, Taproot, MWEB, or Signet as active Defcoin mainnet features. Some inherited Litecoin Core code paths still exist because the app is based on Litecoin Core, but the GUI should avoid presenting unsupported Litecoin-only behavior as a Defcoin feature.

Glossary

DFC The primary Defcoin unit.

Packet 0.001 DFC.

Tock 0.000001 DFC.

Mote 0.00000001 DFC, the smallest displayed unit.

Address A destination string that can receive Defcoin. Treat addresses as exact text; changing one character changes the destination.

Backend The Defcoin node process that validates blocks, stores wallet data, signs transactions, talks to peers, and writes debug.log.

Berkeley DB wallet The legacy wallet.dat database format used by historical Defcoin wallets.

Blockchain The ordered public record of blocks and transactions that full nodes verify.

Confirmation A transaction has one confirmation when it is included in a block. More blocks after that transaction increase its confirmation count.

FQDN Fully qualified domain name, a complete DNS name for a host.

Full node A node that validates blocks and transactions itself instead of trusting a remote wallet service to do it.

HD wallet Hierarchical deterministic wallet that derives future keys from wallet seed material.

Latency Network delay measured from peer ping timing. Lower is usually better.

Mempool Transactions your node knows about that are waiting to be mined into a block.

Peer Another node connected to your node.

Private key Secret wallet material that authorizes spending. Never share it.

PSBT A partially signed transaction package used when creating, reviewing, signing, and broadcasting are separated into multiple steps.

Seed node A DNS host or fixed address used to discover peers when your node starts.

Transaction ID The public identifier for a transaction after it is created or broadcast.

Known Issues And Review Notes

  • Explicit addnode can connect to known IPv6 or non-default-port peers, but automatic discovery still depends on DNS seeds or peer address gossip advertising that exact address and port.
  • The User Agent filter may reduce address-discovery paths if legacy peers identify as Litecoin-style nodes while still relaying useful Defcoin addresses.
  • Nu's newer part is the Qt Quick interface and packaging model. Public distribution should use Developer ID signing/notarization on macOS and a signed installer on Windows.
  • Location, reverse-DNS, and peer labels are best-effort diagnostics, not consensus or identity proof.
